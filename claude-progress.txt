### Feature #209: Implement real AI style analysis with i18n support

**Problem:** The Human Model style analysis was a SIMULATION with hardcoded English values. The analysis needed to:
1. Connect to real AI providers (OpenAI/Anthropic)
2. Read user-uploaded text files
3. Generate style analysis in the correct language (IT/EN)

**Implementation:**

1. **Backend - AI Service (server/src/services/ai-service.ts)** [NEW FILE]
   - Created modular AI service supporting both OpenAI and Anthropic
   - Uses native `fetch` API (no external SDK needed)
   - Auto-detects available providers from environment variables
   - Prompts are fully localized (Italian and English versions)
   - Graceful fallback to mock analysis when no API keys configured
   - Returns structured StyleAnalysisResult: tone, sentence_structure, vocabulary, patterns[]

2. **Backend - Analyze Endpoint (server/src/routes/human-models.ts)**
   - Changed from synchronous setTimeout mock to async AI analysis
   - Accepts `language` parameter from request body (defaults to 'it')
   - Also reads Accept-Language header as fallback
   - Reads all source files from database, combines text
   - Runs analysis asynchronously, updates DB when complete
   - Proper error handling with status updates to 'failed'

3. **Frontend - API Service (client/src/services/api.ts)**
   - `analyzeHumanModel(id, language)` now accepts optional language parameter

4. **Frontend - Human Model Page (client/src/pages/HumanModelPage.tsx)**
   - Imports i18n config to get current language
   - Passes `i18n.language` to analyze API call

5. **Translations (client/src/i18n/locales/*.json)**
   - Added missing keys for analysis results:
     - `humanModel.tone`, `humanModel.sentenceStructure`
     - `humanModel.vocabulary`, `humanModel.patterns`
     - `humanModel.analysisResults`, `humanModel.analysisComplete`
     - `humanModel.analysisCompleteDesc`, `humanModel.noAnalysisData`
   - Both Italian (it.json) and English (en.json) updated

**Key Design Decisions:**
- Mock analysis remains as fallback for development/demo without API keys
- When valid API keys are configured, real AI analysis is used
- Analysis runs asynchronously to not block the HTTP response
- Prompts are crafted to return JSON for easy parsing

**Testing Limitation:**
- Server could not be restarted due to sandbox restrictions
- Implementation verified through code review
- All changes follow established patterns in codebase

### Feature #210: Add LLM Provider database schema

**Objective:** Create the `llm_providers` table to store user AI provider configurations, and update `user_preferences` with provider/model selection fields.

**Implementation:**

1. **Database Schema (server/src/db/database.ts)**
   - Added `llm_providers` table with all required columns:
     - `id` (UUID PRIMARY KEY)
     - `user_id` (FK → users, CASCADE delete)
     - `provider_type` (ENUM: openai, anthropic, google_gemini, open_router, requesty, custom)
     - `display_name` (custom name like "My OpenAI")
     - `api_key_encrypted` (encrypted API key)
     - `api_base_url` (for custom/self-hosted providers)
     - `additional_config_json` (extra config like headers, parameters)
     - `is_active` (boolean, currently active provider)
     - `connection_status` (ENUM: not_tested, connected, failed)
     - `last_test_at` (timestamp of last connection test)
     - `created_at`, `updated_at`
   - Updated `user_preferences` table:
     - Added `selected_provider_id` (FK → llm_providers, SET NULL on delete)
     - Added `selected_model_id` (model ID within selected provider)
   - Added indexes:
     - `idx_llm_providers_user_id`
     - `idx_llm_providers_is_active`
     - `idx_user_preferences_selected_provider_id`

2. **Encryption Utility (server/src/utils/encryption.ts)**
   - Created encryption module using AES-256-GCM
   - Key derivation with PBKDF2 (100,000 iterations, SHA-512)
   - Random salt (64 bytes) and IV (16 bytes) per encryption
   - Authentication tag (16 bytes) for integrity verification
   - Functions:
     - `encrypt(plaintext)` → base64 encoded encrypted string
     - `decrypt(encryptedBase64)` → original plaintext
     - `isEncrypted(value)` → check if value appears encrypted
     - `hash(value)` → SHA-256 one-way hash

3. **Documentation (docs/database-schema.md)**
   - Comprehensive documentation of entire database schema
   - All tables, columns, constraints, indexes
   - Encryption details and security notes
   - Foreign key relationship diagram
   - Index reference table

**Testing:**
- Created test script at `scripts/test-schema.js`
- Verified all tables created correctly
- Verified all columns match specification
- Verified indexes created
- Verified encryption/decryption works correctly

**Security:**
- API keys are encrypted with AES-256-GCM before storage
- Each encryption uses unique salt and IV
- Authentication tags verify data integrity
- Production should set ENCRYPTION_KEY env variable

### Feature #208: Fix Human Model file upload - entity too large + JSON escaping

**Problem:** When uploading files in the Human Model page, two errors occurred:
1. "request entity too large" - server rejected files that were too large
2. "Bad escaped character in JSON" - special characters in the text broke the JSON
Also, the error message shown to the user was incorrect (showed profile name instead of actual error).

**Implementation:**

1. **Backend (server/src/index.ts)**
   - Increased `express.json()` limit from default 100kb to 50MB
   - Increased `express.urlencoded()` limit to 50MB as well

2. **Backend (server/src/routes/human-models.ts)**
   - Added detailed logging for upload requests (model ID, user ID, content length)
   - Added server-side file size validation (10MB max per file)
   - Returns HTTP 413 with clear message for files exceeding limit

3. **Frontend (client/src/pages/HumanModelPage.tsx)**
   - Added client-side file size validation BEFORE reading file (10MB max)
   - Added content size validation before upload (prevents wasted bandwidth)
   - Added toast notifications for success and error cases
   - Added file size display in upload preview (shows KB)
   - Added "Maximum file size: 10 MB" hint in upload dialog
   - Added FileReader error handler

**Key Changes:**
- `server/src/index.ts`: `express.json({ limit: '50mb' })`
- `client/src/pages/HumanModelPage.tsx`: File size checks in both `handleFileRead` and `handleUpload`
- Proper error propagation with toast notifications

**Testing Limitation:**
- Could not perform full browser automation testing due to sandbox restrictions
- Server restart required to apply backend changes (network restrictions in sandbox)
- Frontend changes are code-verified and follow established patterns
