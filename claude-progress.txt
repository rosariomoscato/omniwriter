### Feature #211: Backend API for LLM Provider management

**Objective:** Create REST endpoints for managing user LLM providers with encrypted API keys, connection testing, and model listing.

**Implementation:**

1. **Backend Route (server/src/routes/llm-providers.ts)** [NEW FILE]
   - Complete CRUD implementation for LLM providers
   - API key encryption using `crypto.scryptSync` with AES-256-CBC
   - Connection testing with real API calls
   - Dynamic model fetching with 5-minute cache

2. **Endpoints Implemented:**
   - `GET /api/llm-providers` - List all providers (hides API key, shows last 4 chars)
   - `GET /api/llm-providers/:id` - Get single provider
   - `POST /api/llm-providers` - Create new provider (validates type, encrypts key)
   - `PUT /api/llm-providers/:id` - Update provider (resets connection status on key change)
   - `DELETE /api/llm-providers/:id` - Delete provider (prevents deleting active provider)
   - `POST /api/llm-providers/:id/test` - Test connection (makes real API call)
   - `GET /api/llm-providers/:id/models` - List available models
   - `PUT /api/llm-providers/preferences/llm` - Set preferred provider/model

3. **Provider Types Supported:**
   - `openai` - OpenAI GPT models
   - `anthropic` - Claude models (hardcoded list, no public API)
   - `google_gemini` - Google Gemini models
   - `open_router` - OpenRouter aggregator
   - `requesty` - Requesty aggregator
   - `custom` - Self-hosted (requires base_url)

4. **Security Features:**
   - API keys encrypted before storage using AES-256-CBC
   - Random IV per encryption for uniqueness
   - Keys never returned in API responses
   - Connection status tracked (not_tested, connected, failed)
   - Active provider protection on delete

5. **Frontend API Service (client/src/services/api.ts)**
   - Added `LLMProvider` and `CreateLLMProviderData` interfaces
   - Added all LLM provider API methods:
     - `getLLMProviders()`, `getLLMProvider(id)`
     - `createLLMProvider(data)`, `updateLLMProvider(id, data)`
     - `deleteLLMProvider(id)`
     - `testLLMProvider(id)`
     - `getLLMProviderModels(id)`
     - `updateLLMPreferences(data)`

6. **User Preferences Update (server/src/routes/users.ts)**
   - Updated GET `/api/users/preferences` to include `selected_provider_id` and `selected_model_id`

**Testing Performed:**
- Created test user via API

- Updated provider display name
- Set/unset LLM preferences
- Verified delete protection when provider is active
- Created Anthropic provider, tested models endpoint (returns hardcoded list)

### Feature #212: Multi-provider AI service architecture

**Objective:** Create a modular architecture for communicating with different AI providers in a unified way.

**Implementation:**

1. **Folder Structure (server/src/services/ai/)**
   ```
   server/src/services/ai/
   ├── index.ts              # Unified export
   ├── types.ts              # Common interfaces (ChatMessage, CompletionOptions, etc.)
   ├── BaseProvider.ts       # Abstract base class
   ├── AIServiceFactory.ts   # Factory with lazy loading
   ├── providers/
   │   ├── OpenAIProvider.ts     # OpenAI GPT implementation
   │   ├── AnthropicProvider.ts  # Claude implementation
   │   ├── GeminiProvider.ts     # Google Gemini implementation
   │   ├── OpenRouterProvider.ts # OpenRouter (multi-model aggregator)
   │   ├── RequestyProvider.ts   # Requesty implementation
   │   └── CustomProvider.ts     # Generic provider for self-hosted
   └── __tests__/
       └── providers.test.ts # Unit tests for all providers
   ```

2. **TypeScript Interfaces (types.ts)**
   - `ProviderType`: 'openai' | 'anthropic' | 'google_gemini' | 'open_router' | 'requesty' | 'custom'
   - `ChatMessage`: role + content
   - `CompletionOptions`: model, temperature, maxTokens, etc.
   - `CompletionResponse`: content, model, usage, finishReason
   - `ModelInfo`: model metadata with context window and features
   - `AIError`: Unified error class with type mapping and retryable flag
   - `RetryConfig`: Exponential backoff configuration

3. **BaseProvider Abstract Class**
   - Common interface for all providers
   - `chat(messages, options)`: Chat completion
   - `complete(prompt, options)`: Simple completion
   - `stream(messages, options)`: AsyncGenerator for SSE streaming
   - `countTokens(text)`: Approximate token counting
   - `getModelInfo(modelId)`: Get model details
   - `testConnection()`: Test provider availability
   - `getAvailableModels()`: List supported models
   - `withRetry()`: Exponential backoff retry logic

4. **Provider Implementations**
   - **OpenAIProvider**: GPT-4, GPT-3.5, O1 models with streaming
   - **AnthropicProvider**: Claude 3 family (Opus, Sonnet, Haiku)
   - **GeminiProvider**: Google Gemini 1.5 Pro/Flash with 2M context
   - **OpenRouterProvider**: Multi-model aggregator with OpenAI-compatible API
   - **RequestyProvider**: Alternative aggregator
   - **CustomProvider**: Self-hosted endpoints (Ollama, vLLM, LM Studio)

5. **Error Handling**
   - Unified error mapping for all providers
   - Error types: authentication, rate_limit, context_length, model_not_found, etc.
   - Automatic retry for transient errors (rate limits, timeouts, network)
   - Exponential backoff with configurable delays

6. **Factory Pattern (AIServiceFactory.ts)**
   - `createProvider(type, config)`: Create provider instance
   - `createProviderFromEnv(type)`: Create from environment variables
   - `getFirstAvailableProvider()`: Get first working provider
   - Provider caching for performance

7. **Legacy Compatibility (ai-service.ts)**
   - Wraps new architecture for backward compatibility
   - `analyzeWritingStyle()` still works with new providers
   - `isAIAvailable()` checks any provider availability

**Usage Example:**
```typescript
import { createProvider, getFirstAvailableProvider } from './services/ai';

// Create specific provider
const openai = createProvider('openai', { apiKey: 'sk-...' });
const response = await openai.chat([
  { role: 'user', content: 'Hello!' }
]);

// Stream response
for await (const event of openai.stream(messages)) {
  if (event.type === 'delta') {
    console.log(event.content);
  }
}

// Use first available from environment
const provider = getFirstAvailableProvider();
```

### Feature #210: Add LLM Provider database schema

**Objective:** Create the `llm_providers` table to store user AI provider configurations, and update `user_preferences` with provider/model selection fields.

**Implementation:**

1. **Database Schema (server/src/db/database.ts)**
   - Added `llm_providers` table with all required columns:
     - `id` (UUID PRIMARY KEY)
     - `user_id` (FK → users, CASCADE delete)
     - `provider_type` (ENUM: openai, anthropic, google_gemini, open_router, requesty, custom)
     - `display_name` (custom name like "My OpenAI")
     - `api_key_encrypted` (encrypted API key)
     - `api_base_url` (for custom/self-hosted providers)
     - `additional_config_json` (extra config like headers, parameters)
     - `is_active` (boolean, currently active provider)
     - `connection_status` (ENUM: not_tested, connected, failed)
     - `last_test_at` (timestamp of last connection test)
     - `created_at`, `updated_at`
   - Updated `user_preferences` table:
     - Added `selected_provider_id` (FK → llm_providers, SET NULL on delete)
     - Added `selected_model_id` (model ID within selected provider)
   - Added indexes:
     - `idx_llm_providers_user_id`
     - `idx_llm_providers_is_active`
     - `idx_user_preferences_selected_provider_id`

2. **Encryption Utility (server/src/utils/encryption.ts)**
   - Created encryption module using AES-256-GCM
   - Key derivation with PBKDF2 (100,000 iterations, SHA-512)
   - Random salt (64 bytes) and IV (16 bytes) per encryption
   - Authentication tag (16 bytes) for integrity verification
   - Functions:
     - `encrypt(plaintext)` → base64 encoded encrypted string
     - `decrypt(encryptedBase64)` → original plaintext
     - `isEncrypted(value)` → check if value appears encrypted
     - `hash(value)` → SHA-256 one-way hash

3. **Documentation (docs/database-schema.md)**
   - Comprehensive documentation of entire database schema
   - All tables, columns, constraints, indexes
   - Encryption details and security notes
   - Foreign key relationship diagram
   - Index reference table

**Testing:**
- Created test script at `scripts/test-schema.js`
- Verified all tables created correctly
- Verified all columns match specification
- Verified indexes created
- Verified encryption/decryption works correctly

**Security:**
- API keys are encrypted with AES-256-GCM before storage
- Each encryption uses unique salt and IV
- Authentication tags verify data integrity
- Production should set ENCRYPTION_KEY env variable

### Feature #208: Fix Human Model file upload - entity too large + JSON escaping

**Problem:** When uploading files in the Human Model page, two errors occurred:

**Implementation:**

1. **Backend (server/src/index.ts)**
   - Increased `express.json()` limit from default 100kb to 50MB

2. **Backend (server/src/routes/human-models.ts)**
   - Added detailed logging for upload requests (model ID, user ID, content length)
   - Added server-side file size validation (10MB max per file)
   - Returns HTTP 413 with clear message for files exceeding limit

3. **Frontend (client/src/pages/HumanModelPage.tsx)**
   - Added client-side file size validation BEFORE reading file (10MB max)
   - Added content size validation before upload (prevents wasted bandwidth)
   - Added toast notifications for success and error cases
   - Added file size display in upload preview (shows KB)
   - Added "Maximum file size: 10 MB" hint in upload dialog
   - Added FileReader error handler

**Key Changes:**
- `server/src/index.ts`: `express.json({ limit: '50mb' })`
- `client/src/pages/HumanModelPage.tsx`: File size checks in both `handleFileRead` and `handleUpload`
- Proper error propagation with toast notifications

**Testing Limitation:**
- Could not perform full browser automation testing due to sandbox restrictions
- Server restart required to apply backend changes (network restrictions in sandbox)
- Frontend changes are code-verified and follow established patterns
